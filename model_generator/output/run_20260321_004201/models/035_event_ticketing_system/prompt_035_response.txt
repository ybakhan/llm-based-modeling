CONSTRUCTS_INVOLVED: Purchase Ticket, Validate Payment Details
EXPLANATION: The use case "Validate Payment Details" is an inclusion use case that is included by only one use case ("Purchase Ticket"). "Validate Payment Details" is not associated with any actor directly — it has no independent actor relationship. It does not include, extend, generalize, or get extended by any other use case. It represents a sub-step of purchasing a ticket rather than a complete, independently valuable service observable to a user. This is a textbook case of functional decomposition where a single use case's internal behavior has been improperly broken out into a separate inclusion use case that provides no standalone value.

REFACTORING_RATIONALE: The inclusion use case "Validate Payment Details" must be merged back into "Purchase Ticket" because it serves only as an internal step of that use case and provides no complete service on its own. By applying Drop Functional Decomposition, we remove "Validate Payment Details" and the <<include>> relationship between "Purchase Ticket" and "Validate Payment Details". The payment validation logic becomes an internal step within "Purchase Ticket," which already represents the complete service from the customer's perspective. This reduces the model by one use case and one include relationship.
