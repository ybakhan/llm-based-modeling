[ANTIPATTERN 1]
ANTIPATTERN_DETECTED: Functional Decomposition: Using the include relationship
INSTANCE_COUNT: 1

[INSTANCE 1]
CONSTRUCTS_INVOLVED: Perform Inspection, Record Measurements
EXPLANATION: The use case "Record Measurements" is an inclusion use case that is included by only one use case ("Perform Inspection"). "Record Measurements" is not associated with any actor directly, does not include or extend any other use case, is not extended by any use case, and is not involved in any generalization relationship. It represents a sub-step of the inspection process rather than a complete, independently valuable service observable to an actor. This is a textbook example of functional decomposition via the include relationship — the designer has broken "Perform Inspection" into a procedural sub-step that has no standalone value.

REFACTORING_RATIONALE: The inclusion use case "Record Measurements" must be merged back into its sole base use case "Perform Inspection" because it does not provide a complete service to any actor on its own. By applying the "Drop Functional Decomposition" refactoring, we remove "Record Measurements" and the <<include>> relationship between "Perform Inspection" and "Record Measurements". The behavior of recording measurements becomes an internal part of "Perform Inspection". This eliminates the functional decomposition antipattern while preserving all meaningful system functionality.
CONSTRUCT_COUNT: 10
